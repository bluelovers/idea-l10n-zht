根據數據流分析，報告始終違反為 null 性約定、可能拋出異常或只是冗餘的程式碼結構。

範例：

if (array.length < index) {
  System.out.println(array[index]);
} // 陣列索引始終超出界限

if (str == null) System.out.println("str is null");
System.out.println(str.trim());
// 最後一個語句可能會拋出 NPE

@NotNull
Integer square(@Nullable Integer input) {
    // 違反方法約定
    return input == null ? null : input * input;
}

檢查行為可能由許多註解控制，例如為 null 性註解、@Contract 註解、@Range 註解等。

組態檢查：

  • 使用對可 null 值的方法/欄位/參數建議使用 @Nullable 註解選項可以在以下情況下發出警告：null 值作為實參傳遞給具有非註解參數的方法時，存儲到非註解欄位時，或從非註解方法返回時。 在這種情況下，該檢查將建議傳播 @Nullable 註解。 您還可以使用組態註解按鈕組態可 null 性註解。
  • 使用將未註解的成員和參數視為 @Nullable 選項可假設非註解成員可以為 null，因此不得在非 null 上下文中使用。
  • 使用報告使用 null 文字實參用法的非 null 必需參數>選項可以報告不能為 null（例如在方法體中立即取消參照），但存在傳遞 null 文字的呼叫點的方法參數。
  • 使用報告始終返回非 null 值的可 null 方法選項，可報告註解為 @Nullable 但始終返回非 null 值的方法。 在這種情況下，建議您將註解更改為 @NotNull。
  • 使用忽略 assert 語句選項可控制檢查如何處理 assert 語句。 預設情況下，該選項被停用，這意味著斷言視為被執行（-ea 模式）。 如果啟用該選項，斷言將被完全忽略（-da 模式）。
  • 使用報告僅在某些程式碼路徑上發生的問題選項可控制是否報告僅在某些程式碼路徑上發生的問題。 如果停用此選項，則不會報告 exception is possible 等警告。 該檢查只會報告類似 exception will definitely occur 的警告。 這種模式可以大大減少誤報的數量，特別是如果程式碼沒有一致地使用可 null 性和約定註解進行註解。 這就是為什麼它對於尋找傳統程式碼庫中最重要的問題很有用。

在 IntelliJ IDEA 2022.3 版本之前，此檢查為“常數條件和異常”檢查的一部分。 現在，它被拆分成兩個檢查：“常量值”和“為 null 性和數據流問題”。
